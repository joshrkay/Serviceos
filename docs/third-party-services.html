AWS — Amazon Web Services

ECS/Fargate · RDS Postgres · S3 · SQS · ECR · Secrets Manager · CloudWatch

Partial — CDK missing 4 stacks
What's in CDK today
  • VPC (public/private/isolated subnets, 2 AZs)
  • ECS Cluster + Fargate task definition
  • ECR repository (per env)
  • ALB + HTTP listener + health check
  • CloudWatch log groups (/serviceos/{env}/api)
  • CDK app deploys dev/staging/prod stacks
What's missing from CDK
  • RDS Postgres (no database stack at all)
  • S3 bucket (no file storage stack)
  • SQS queues + DLQs (no queue stack)
  • Secrets Manager placeholders
  • HTTPS on ALB (only HTTP today — security issue)
  • ECR image scanning enabled
  • Auto-scaling policy for ECS service
1
Create / log into AWS account

Use aws.amazon.com. For a new account, set up MFA on root immediately and create an IAM admin user for day-to-day use. Enable billing alerts.

2
Create a deployment IAM role for GitHub Actions

The CI/CD workflow uses secrets.AWS_ROLE_ARN via OIDC (no long-lived keys). Create an IAM role with a trust policy for token.actions.githubusercontent.com and permissions for CDK deploy (CloudFormation, ECS, ECR, S3, RDS, SQS, SecretsManager, IAM). Add the role ARN to GitHub repo → Settings → Secrets → AWS_ROLE_ARN.

3
Bootstrap CDK in each region

Run once per AWS account/region: npx cdk bootstrap aws://ACCOUNT_ID/us-east-1. Sets up CDK toolkit stack (S3 bucket for assets, IAM roles).

4
Add missing CDK stacks (RDS, S3, SQS, Secrets)

The current platform-stack.ts is missing these. They need to be added before any deploy will work end-to-end. Add to infra/src/stacks/: data-stack.ts (RDS + S3), queue-stack.ts (SQS + DLQs), secrets-stack.ts (Secrets Manager placeholders).

5
Add AWS account IDs to infra/src/config.ts

The account field in EnvironmentConfig is optional and unset. Each environment needs its AWS account ID: account: '123456789012'. Dev/staging/prod can share one account or be separate accounts.

6
Fix HTTPS on the ALB

Currently only HTTP port 80. Before users touch it, add an ACM certificate + HTTPS listener (port 443) and redirect HTTP → HTTPS. Register a domain (Route 53 or external) and request a cert via ACM.

Clerk

Authentication · Session management · Tenant bootstrap webhooks

Partial — JWT verify is a stub
What's wired up
  • Token parsing + payload extraction
  • Tenant ID + role read from JWT claims
  • Tenant bootstrap logic (idempotent)
  • Auth middleware (requireAuth, requirePermission)
  • RBAC role/permission model complete
What's missing / needs fixing
  • JWT signature NOT verified — critical security bug
  • JWT template with tenant_id + role claims not configured in Clerk dashboard
  • Webhook handler for user.created event
  • Clerk webhook signing secret not wired
  • Frontend: ClerkProvider + sign-in/up components
1
Create Clerk account + application

Sign up at clerk.com. Create an application called "ServiceOS". Create separate dev and prod instances (Clerk gives you one per environment).

2
Grab your API keys

Dashboard → API Keys. Copy Secret key (sk_live_... or sk_test_...) and Publishable key (pk_live_... / pk_test_...). The secret key goes in Secrets Manager. The publishable key is safe in env vars / frontend.

3
Configure a JWT template with custom claims

Dashboard → JWT Templates → New template → "Blank". Add two custom claims that the API reads from every token: tenant_id (from user public metadata) and role (from user public metadata). The API currently reads these as payload.tenant_id and payload.role. Without this, every authenticated user will have no tenant context.

4
Configure webhook endpoint

Dashboard → Webhooks → Add endpoint. URL: https://your-api-domain/webhooks/clerk. Subscribe to: user.created, user.updated. Copy the Signing secret — this goes in Secrets Manager as CLERK_WEBHOOK_SECRET. This triggers the tenant bootstrap flow.

5
Fix JWT signature verification in code

The current decodeClerkToken() in auth/clerk.ts decodes but never verifies the signature — the _secretKey parameter is unused. Must be fixed to use Clerk's JWKS endpoint (https://<your-clerk-domain>/.well-known/jwks.json) to verify the RS256 signature before trusting any claim.

Environment Variables

CLERK_SECRET_KEYBackend secret key (sk_live_...). Secrets Manager in staging/prod. Never expose client-side.required
CLERK_PUBLISHABLE_KEYFrontend publishable key (pk_live_...). Safe to include in frontend bundle.required
CLERK_WEBHOOK_SECRETWebhook signing secret — needed to verify user.created events that trigger tenant bootstrap. Not yet in config schema — needs to be added.required
🚨 Critical fix required before any production use: auth/clerk.ts decodeClerkToken() does not verify JWT signatures. Any attacker can craft a token with any tenant_id or role claim and it will be accepted. Must implement JWKS-based RS256 verification.

Sentry

Error tracking · Exception monitoring · Performance tracing

Stub — console fallback only
What's wired up
  • SentryClient interface + initSentry() function
  • No-op client when DSN is absent (safe for dev)
  • captureException / captureMessage signatures ready
What's missing
  • Actual @sentry/node SDK not initialized
  • DSN not yet stored or injected
  • No PII scrubbing configured
  • Express request handler middleware not added
  • No release/version tagging
1
Create Sentry account + project

Sign up at sentry.io. Create a new project → Platform: Node.js / Express. Name: serviceos-api.

2
Copy DSN

Settings → Projects → serviceos-api → Client Keys (DSN). Looks like https://abc123@o456.ingest.sentry.io/789. Create separate projects (and DSNs) for dev/staging/prod or use environments within one project.

3
Replace the stub with real SDK initialization

In monitoring/sentry.ts, replace the logging fallback with Sentry.init({ dsn, environment, release, tracesSampleRate: 0.1 }). Add Sentry.requestHandler() middleware before routes and Sentry.errorHandler() after. @sentry/node is already in dependencies.

4
Configure PII scrubbing

In Sentry init, add beforeSend to strip phone numbers, email addresses, and tenant-identifying data from error payloads. HVAC customer data must not appear in Sentry.

Environment Variables

SENTRY_DSNProject DSN from Sentry dashboard. Optional — if absent, Sentry is silently disabled (no-op client).optional
Note: Sentry is optional for local dev. Set SENTRY_DSN in staging and prod Secrets Manager. Leaving it empty causes the no-op client to be used — no errors, just silence.

OpenAI

Whisper (voice transcription) · GPT models (LLM gateway — Phase 1+)

Stub — TranscriptionProvider not implemented
What's wired up
  • TranscriptionProvider interface defined
  • Transcription worker calls provider.transcribe()
  • AiTaskType enum includes all task types
  • ai_runs table ready to log provider calls
What's missing
  • Concrete OpenAI Whisper implementation of TranscriptionProvider
  • OPENAI_API_KEY not in config schema
  • LLM gateway (Phase 1) — only needed after P0
  • Model tier routing config (lightweight/standard/complex)
1
Create OpenAI account + get API key

Go to platform.openai.com → API keys → Create new key. Name it serviceos-prod. Store in AWS Secrets Manager — never in .env for staging/prod.

2
Set up usage limits + billing

Platform → Billing → Set a monthly spend limit (e.g. $50 dev, $500 staging) to prevent runaway costs during development. Enable usage alerts.

3
Implement OpenAITranscriptionProvider

Create voice/openai-transcription.ts implementing TranscriptionProvider. Call POST https://api.openai.com/v1/audio/transcriptions with model whisper-1. The audio URL from S3 needs to be fetched and streamed — Whisper accepts file uploads, not URLs directly.

4
Add OPENAI_API_KEY to config schema

Currently missing from shared/config.ts. Add: OPENAI_API_KEY: z.string().min(1).optional(). Make it required in staging/prod via the secrets resolver.

Environment Variables

OPENAI_API_KEYOpenAI API key. Used for Whisper transcription (Phase 0) and GPT models (Phase 1+). Secrets Manager in staging/prod. Not yet in config schema — needs to be added.required
Alternative for transcription: AWS Transcribe is another option that would keep everything within AWS (no extra vendor). Downside is higher latency and lower accuracy vs Whisper. If you go AWS Transcribe, you can skip this service account entirely for Phase 0 — it uses your existing AWS IAM role.
