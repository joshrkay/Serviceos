## Key technical decisions
- **Standing instructions shape drafts, never approvals** — injected only into drafting prompts (never the intent classifier, never `decideInitialStatus`), capped (≤20 stored, ≤5 injected, deterministic scope match), pinned by a guard test that `_meta.appliedStandingInstructions` cannot change proposal status. (Alternative — instruction-driven auto-approval — rejected: breaks the trust spine.)
- **Conversational owner assistant reuses what already ships**: browser mic → Deepgram via the existing `useDeepgramDictation` grant-token path (`POST /api/voice/stream-token`), replies via existing `POST /api/assistant/chat`, browser `useTTS` for v1, client-side barge-in. No new WS surface, no schema change. Voice approve/reject stays EXCLUDED in-app (RV-071/225 posture kept). (Alternative — wiring the customer-calling FSM `inapp-adapter` — rejected: wrong persona for the owner.)
- **Spanish = static language at call start + explicit switch only** (finish+reopen Deepgram with `language:'es'`; flap-guard max 2 switches; Spanish fillers; bilingual emergency keywords scanned unconditionally). (Alternative — per-utterance auto-switching — rejected: flapping risk on the revenue path.)
- **Autonomous booking is a scoped lane, not a policy change**: new pure evaluator module + one narrow optional-input branch in `decideInitialStatus`; only `create_appointment`/`create_booking` (capture class), only inbound-receptionist call sites, only when tenant opted in AND confidence ≥ dedicated threshold (default 0.95, floor 0.90) AND resolution clean AND held slot in business hours AND no vulnerability/emergency/negotiation flag. Instant customer confirm after the existing 5s undo window; owner gets one-tap HMAC UNDO (30 min) that cancels + sends a fixed-template apology. (Alternative — treating `supervisorPresent` as true — rejected: leaks permissiveness into all capture types.)
- **One taxonomy train**: all new intents (`create_invoice_schedule`, `respond_to_review`, `create_standing_instruction`) land as one coordinated `INTENT_TAXONOMY_VERSION` 1.1.0→1.2.0 bump + one catalog update, not three.
