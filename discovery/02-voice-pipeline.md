## Summary

Rivet's voice stack is unusually well-engineered at the resilience layer ŌĆö replay-protected webhooks, a per-session promise-chain lock shared by both transports, a realtime health circuit with establish-then-die semantics, mid-call REST degrade from Media Streams back to Gather, bounded apology/reprompt ladders (VOX-35c, A3), durable dropped-call recovery, and complete Twilio signature coverage including the WS upgrade. The FSM is a clean pure reducer with side-effects-as-data. However, the *latency* story misses the 0.5ŌĆō1.0s end-of-speechŌåÆresponse target by roughly 2ŌĆō3x on both transports: Deepgram endpointing is pinned at 600ms, the intent LLM adds 400ŌĆō1,500ms, and ŌĆö critically ŌĆö the 250ms filler-audio timer is armed only when TTS rendering starts, i.e. *after* the LLM call completes, so filler masks TTS startup but not the dominant LLM thinking gap (despite an inline comment claiming otherwise). The single most alarming find: the ElevenLabs streaming URL never pins `output_format=pcm_16000` while the adapter treats every chunk as raw PCM16@16k ŌĆö if ElevenLabs' documented default (mp3) applies, live Media Streams agent audio is static, and *no automated gate would catch it* (Layer 2 uses OpenAI TTS + an unsigned in-process server; the daily real-call smoke only asserts a proposal row, which succeeds even if the caller-side audio is garbage). Scale is a hard single-replica ceiling by design (in-memory `VoiceSessionStore`, `numReplicas = 1` in railway.voice.toml); the 1,000-concurrent validation and the voice capacity table are both "TBD," and the voice load-test harness as written cannot actually exercise the production adapter (unsigned upgrade ŌåÆ 403; `start` frame lacks `callSid` ŌåÆ 1008 close). Silence handling is asymmetric: Gather-mode silent callers likely get hung up (no `actionOnEmptyResult`), Media-Streams silent callers wait indefinitely (no per-turn silence timer). Vapi is live but deliberately shallow (session recording + activation only ŌĆö documented in docs/solutions).

## Findings

**T2-F01 | ElevenLabs streaming output format never pinned to PCM ŌĆö live agent audio may be static | Bug (default-format now verified) | Critical | Effort S | Confidence High**

> **Update 2026-07-18:** ElevenLabs' API reference confirms the stream-input WebSocket `output_format` parameter **defaults to `mp3_44100`** when omitted (https://elevenlabs.io/docs/api-reference/text-to-speech/v-1-text-to-speech-voice-id-stream-input; corroborated by elevenlabs-python issue #251, "Input streaming ŌĆö always return in MP3 format"). Since the code omits the parameter and the consumer decodes chunks as raw PCM16@16k, the realtime path emits MP3 bytes into a PCM decoder ŌĆö near-certain static whenever Media Streams mode is active. Remaining unknowns are only (a) whether `voice_realtime`/Media Streams is enabled for any prod/staging tenant and (b) a final ear-check; the key needed for an empirical probe exists only in Railway, not in this audit environment or GitHub secrets.
Evidence: `ai/tts/elevenlabs-stream.ts:66-67` builds `wss://.../stream-input?model_id=...&xi-api-key=...` with **no `output_format`**; ElevenLabs' documented default for stream-input is mp3. The consumer treats every frame as raw PCM16@16k (`tts-provider.ts:44-49` "PCM 16-bit... @ 16 kHz"; `mediastream-adapter.ts:2325-2344` mu-law-encodes bytes directly). The team clearly knows the format matters: the REST filler renderer pins `?output_format=pcm_16000` (`scripts/render-fillers.ts:49`) and the buffered path has an `isRawPcmContentType` guard specifically because "compressed audio... produces inaudible static with no error" (mediastream-adapter.ts:629-643) ŌĆö but the *streaming* path, the one actually used on every live turn, has no such guard and no format pin. **No gate covers this**: Layer 2 uses `OpenAiTtsProvider` + mp3 decode for the *caller* side with `authTestMode` bypassing the real server; `voice-smoke-real.ts` only asserts a proposal row lands ŌĆö that passes even if the caller hears static, because the scripted caller leg speaks regardless of what it hears. Plan: add `output_format=pcm_16000` to the WS URL; add a magic-byte/energy sanity check on the first chunk; add one live-audio assertion (Whisper-transcribe the agent leg in the daily smoke).

**T2-F02 | Filler audio masks TTS startup, not the LLM thinking gap ŌĆö dead air Ōēł1.5ŌĆō2.9s per turn | Latency design gap | High | Effort M | Confidence High**
Evidence: `onTranscriptEvent` awaits `speechTurn` (the LLM call, `mediastream-adapter.ts:1243-1250`) *before* `emitSideEffects` ŌåÆ `runTurnWithFiller`, where the 250ms filler timer is first armed (mediastream-adapter.ts:2086, 2114-2139). So during endpointing (600ms) + LLM (400ŌĆō1,500ms under the 1.5s lightweight deadline) the caller hears nothing; the filler only covers ElevenLabs TTFB. The inline comment at :2354 ("filler audio fills the LLM thinking gap") describes the intended design, not the implemented one. Combined with T2-F11, typical end-of-speechŌåÆfirst-audio is ~1.4ŌĆō2.9s vs the 0.5ŌĆō1.0s target (the repo's own SLO reference is "1.5s voice-turn SLO", elevenlabs-stream.ts:9). Plan: arm the filler timer at final-transcript time (where `turnLatencyStartMs` is already stamped, :1238), and/or start TTS on the first sentence of a streamed LLM response.

**T2-F03 | Gather mode: silent caller is hung up, not reprompted (`actionOnEmptyResult` missing) | Bug | High | Effort S | Confidence High**
Evidence: `buildTwiML` emits `<Gather input="speech" speechTimeout="auto" ... action=... />` as the **last** verb with no `timeout` and no `actionOnEmptyResult` (`twilio-adapter.ts:676-679`). Twilio's default `actionOnEmptyResult=false` means a no-speech timeout falls through the Gather to the end of the document ŌåÆ Twilio hangs up. The empty-`SpeechResult` ŌåÆ `confidence_low` reprompt path (`twilio-adapter.ts:1845-1855`) is therefore mostly unreachable via silence. The team uses the correct pattern elsewhere ŌĆö `buildCallbackGatherTwiml` appends a trailing `<Redirect>` explicitly "on silence so a call_me_back task is still scheduled" (routes/telephony.ts:963-981) ŌĆö but the main loop lacks it. Caller experience: pause to check your calendar mid-call and the AI hangs up on you. Plan: add `actionOnEmptyResult="true"` or a trailing `<Redirect>` to the gather action.

**T2-F04 | Hard single-replica ceiling; 1,000 concurrent is not reachable and never validated | Scale limit | High | Effort XL | Confidence High**
Evidence: sessions, CallSid index, locks, `pendingTransferTwiml`, `callerIdBySession`, low-confidence streaks, and `WhisperCache` are all process-local (`voice-session-store.ts:314-342`, `twilio-adapter.ts:735-765`, app.ts:3060); `railway.voice.toml` pins `numReplicas = 1` and itself warns replicas need "cluster-wide state." Redis exists only for *event* fan-out (`redis-voice-event-transport.ts`) and the connection-cap registry, not session state. Capacity is unmeasured: both the HTTP U6 mixed-1000 row and the voice per-instance ceiling row are literally "TBD" (`docs/runbooks/voice-capacity.md`). At high call counts a single Node process does per-20ms-frame JS mu-law decode + 8kŌåö16k resample per call (`mulaw-codec.ts`) plus one Deepgram WS per call and one ElevenLabs WS *per turn* ŌĆö CPU will be the first wall, long before 1,000. Deepgram/ElevenLabs account concurrency limits are nowhere modeled or rate-limited. Plan: run the staging capacity test (blocked by T2-F06), record the ceiling, then execute the documented Redis-session phase before any multi-replica voice deploy.

**T2-F05 | Media Streams: no silence reprompt ŌĆö a silent caller waits forever | UX gap | High | Effort S | Confidence High**
Evidence: the only inactivity mechanism is the 30-minute `audioIdleTimer` re-armed on every inbound `media` frame (mediastream-adapter.ts:1136-1148, 2443-2455) ŌĆö and Twilio streams silence frames continuously on a live call, so it never fires mid-call. There is no per-turn "no final transcript in N seconds ŌåÆ reprompt" timer (grep for silence/reprompt/VAD confirms only the STT-confidence ladders). Gather at least *attempts* the `confidence_low` reprompt (see T2-F03). Caller experience: say nothing after the greeting and the line just sits there. Plan: arm a 6ŌĆō10s no-transcript timer after each agent turn that dispatches the existing `confidence_low` reprompt path.

**T2-F06 | Voice load-test harness cannot exercise the real adapter ŌĆö staging capacity run would 403/1008 | Verification gap | High | Effort S | Confidence High**
Evidence: `scripts/voice-load-test.ts:66,71` opens the WS with no `X-Twilio-Signature` (production upgrade rejects 403, `twilio-mediastream-server.ts:155-167`) and sends `start` with only `streamSid` ŌĆö `isValidStartFrame` requires `callSid` (mediastream-adapter.ts:2640-2646), so even a signed connection closes 1008 `invalid_start_payload`/`unknown_callsid`. The self-check's mock server accepts it (`voice-load-selfcheck.ts`), so the H5 quality-gate ack (`voice_capacity_provenance: "harness-self-check"`) proves only the harness. The documented pre-cutover staging run (`docs/runbooks/voice-capacity.md`) will fail or measure nothing real. Plan: teach the harness to sign upgrades (it has the staging auth token in CI secrets) and to first hit `/voice` to mint a real session/CallSid.

**T2-F07 | Tenant WS-cap rejection (50) drops the call with no Gather fallback | Failure-path gap | Medium | Effort S | Confidence High**
Evidence: `perTenantMax: 50` (`ws/connection-registry.ts:26-28`); on lease failure the adapter does `closeWs(1013, 'tenant_connection_cap')` (mediastream-adapter.ts:865-871) ŌĆö pre-establishment, so `attemptDegradeToGather` is never tried and the `<Connect>` document ends ŌåÆ Twilio hangs up. The 51st concurrent caller of a busy tenant hears the greeting TwiML never start and the call die. Plan: on cap rejection, attempt the same REST redirect to `/voice/gather-fallback` used for Deepgram failures (Gather has no per-tenant cap).

**T2-F08 | /gather turns are serialized but not idempotent ŌĆö Twilio retry can double-dispatch an utterance | Idempotency gap | Medium | Effort M | Confidence Medium**
Evidence: `/voice` has CallSid replay protection (`establishInboundSession`, twilio-adapter.ts:1027-1030) but `handleGather` (`:1759-1770`) only wraps in `withSessionLock` ŌĆö a Twilio 15s-timeout retry of the same POST would run the full turn twice back-to-back (two classifier calls, potentially two proposals; FSM state guards mitigate but don't cover the lookup/approval side paths). Exposure requires a turn >15s, plausible only when the gateway is degraded. Plan: dedupe on (sessionId, SpeechResult hash, ~20s window) or Twilio's idempotency token header.

**T2-F09 | Barge-in triggers on any non-empty interim ŌĆö background speech/noise clears agent audio; Gather mode has no barge-in at all | Turn-taking | Medium | Effort M | Confidence High**
Evidence: any interim while `agentSpeaking` calls `bargeIn()` unconditionally (mediastream-adapter.ts:1170-1173) ŌĆö no min-confidence, no min-word-count, no min-duration; a TV or side conversation transcribed by Nova-3 will cut the agent off mid-sentence and flush the queue. (The mechanics themselves are correct and fast: abort controller + Twilio `clear` + queue clear.) On the Gather path, `<Say>` verbs are not interruptible ŌĆö callers must wait out the full agent line; inherent to Gather but worth stating as the fallback experience. Plan: require Ōēź2 words or confidence Ōēź0.5 on the interim before barge-in; optionally resume-on-false-positive.

**T2-F10 | STT/TTS API keys passed in WebSocket URL query strings | Security hygiene | Medium | Effort S | Confidence High**
Evidence: Deepgram `...&token=${this.apiKey}` (`transcription-providers.ts:388`); ElevenLabs `...&xi-api-key=${apiKey}` (`elevenlabs-stream.ts:67`). URLs leak into provider access logs, egress proxies, and any error message that stringifies the URL. Both vendors support header/subprotocol auth. Related: could not verify from the repo that Deepgram *accepts* query-param `token=` auth at all ŌĆö if it doesn't, every realtime open fails and the health circuit silently pins all traffic to Gather (which would look like "works, but never realtime"). Plan: move to header/subprotocol auth; check staging logs for `deepgram_open_failed` frequency.

**T2-F11 | 600ms Deepgram endpointing is a fixed floor on every turn; no early/speculative processing | Latency | Medium | Effort M | Confidence High**
Evidence: `endpointing = options.endpointingMs ?? 600` (`transcription-providers.ts:338`) and no call site ever passes `endpointingMs` (grep). No `utterance_end_ms`, no interim-based speculative LLM start (interims are used only for barge-in and the emergency scan, mediastream-adapter.ts:1170-1177). 600ms is a defensible anti-premature-final choice, but combined with T2-F02 it guarantees the target budget is unreachable. Plan: tune to 400ŌĆō500ms, or keep 600ms and start classification speculatively on a stable interim, discarding on mismatch.

**T2-F12 | TTS mid-stream failure recovery step 1 is dead code with ElevenLabs; filler cache will happily stream mp3 as PCM | Failure-path quality | Medium | Effort S | Confidence High**
Evidence: `recoverTurnAfterStreamFailure` first tries buffered `synthesize()` but ElevenLabs REST returns `audio/mpeg` (`tts-provider.ts:153,171-173`), which the PCM guard then drops (mediastream-adapter.ts:2277-2291) ŌĆö so actual recovery is "filler clip or silence." Separately, `FillerAudioCache` loads `.mp3`/`.bin` files as raw PCM by design ("legacy render runs," `filler-audio-cache.ts:29-33`) ŌĆö an operator dropping an mp3 render into the fillers dir ships static. Also: fillers are an optional Docker build step; without the build-arg key the image ships silent (Dockerfile:53-64) ŌĆö fine, but it widens the T2-F02 dead-air gap. Plan: give ElevenLabs REST a `pcm_16000` variant for the recovery path; drop the `.mp3` cache fallback.

**T2-F13 | Nothing in CI proves the realtime path against real Twilio/Deepgram/ElevenLabs | Verification gap | Medium | Effort M | Confidence High**
Evidence: PR + nightly gates are Layer 1 (mocked LLM, cassettes, in-memory repos ŌĆö pr-checks.yml:77-123, voice-quality-nightly.yml:11-13 "Layer 1 is memory-only"). Pre-deploy Layer 2 is real LLM + real *OpenAI* TTS through an in-process server with `authTestMode: true` (signature bypass, `twilio-mediastream-server.ts:66-78`). The daily `voice-smoke-real` is genuinely real (Twilio ŌåÆ staging ŌåÆ proposal in DB) but asserts only the DB row ŌĆö agent-audio audibility, barge-in, degrade-to-Gather, and the whisper transfer path have no automated real-world proof. `verify:telephony` checks config booleans only. Plan: extend the daily smoke to record the agent leg and Whisper-assert its transcript (the codebase already has all the pieces).

**T2-F14 | Media Streams greeting starts only after Deepgram opens + 2 DB lookups + persona resolve ŌĆö ~1ŌĆō2s of post-answer silence | Latency | Low | Effort M | Confidence Medium**
Evidence: `handleStart` sequence: escalation settings resolve ŌåÆ keywords ŌåÆ language resolver ŌåÆ Deepgram open (Ōēż5s bound) ŌåÆ `initializeSession` under lock, which awaits disclosure/consent ledger, `identifyCaller` (Pg), persona resolve, timeline log, owner push before the greeting TTS begins (mediastream-adapter.ts:885-1006, twilio-adapter.ts:1090-1283). Gather mode's greeting is faster to first audio (Twilio Polly starts on TwiML receipt). Plan: parallelize the pre-greeting lookups; consider playing a canned PCM greeting stub while bootstrap completes.

**T2-F15 | Twilio security posture is solid | Positive | ŌĆö | ŌĆö | Confidence High**
All webhook routes sit behind `requireTwilioSignature` (routes/telephony.ts:325-329), fail-closed on missing token (twilio-signature.ts:90-96), with `PUBLIC_API_URL` canonical-URL reconstruction; recording + voicemail-status mount their own signature-checked sub-routers *before* the shared parser (routes/telephony.ts:276-316); the WS upgrade verifies HMAC over the upgrade URL with draining rejection and a socket-error swallow to prevent process-killing resets (twilio-mediastream-server.ts:126-168); the whisper PII route is signature-gated (app.ts:3795-3802); Vapi verifies per-tenant secrets fail-closed with the receipt/processed idempotency pattern (vapi/webhook.ts:16-30, 52-63). The `authTestMode` escape hatch is deliberately not env-driven and is lint-guarded (VQ2-007). One residual: media-stream tenant binding trusts the in-memory `findByCallSid` mapping plus an *optional* `customParameters.tenantId` cross-check (mediastream-adapter.ts:843-851) ŌĆö acceptable at 1 replica since the signed upgrade gates entry.

**T2-F16 | DTMF is a declared event type with zero producers | Dead code / accessibility gap | Low | Effort S (delete) or L (implement) | Confidence High**
`dtmf_received` exists only in `ai/agents/customer-calling/types.ts:39`; no Gather accepts `input="dtmf"` and the media path never decodes digits. Callers who can't be understood (T2-F03/F05 ladders) have no keypad escape to "press 0 for a human."
